# 案件事实-关系图谱 技术设计说明

## 1. 文档目标
本文档用于将现有 PRD 和图谱模型进一步细化为可研发落地的技术方案，重点回答以下问题：

- 卷宗如何进入处理链路
- 非结构化文本如何转为结构化图谱
- 实体、关系、来源、版本如何存储
- 人工校核如何覆盖自动抽取结果
- 首版如何以可控复杂度交付

## 2. 总体技术目标
首版采用“文书解析 + 规则与模型协同抽取 + 三元组结构化存储 + 图谱查询展示 + 人工校核回写”的分层方案。目标不是一次性做到完全自动化，而是优先保证以下能力：

- 抽取结果稳定
- 来源可回溯
- 关系可校核
- 版本可管理
- 后续可平滑扩展到全案件类型

## 3. 总体架构

```mermaid
flowchart LR
  upload[卷宗上传] --> ingest[接入与编目]
  ingest --> parse[文书解析与切片]
  parse --> ocr[OCR与文本清洗]
  ocr --> extract[实体与事实抽取]
  extract --> resolve[实体归一化与消歧]
  resolve --> mapping[关系映射与三元组生成]
  mapping --> provenance[来源挂载与置信度计算]
  provenance --> store[结构化存储]
  store --> query[图谱查询服务]
  query --> ui[图谱前端]
  ui --> review[人工校核]
  review --> feedback[规则样本沉淀]
  feedback --> extract
```

## 4. 分层设计

### 4.1 接入层
负责接收案件下的卷宗材料，并建立处理上下文。

输入对象：
- 案件基础信息
- 上传的文书文件
- 文书元数据，如文书类型、上传人、上传时间

输出对象：
- `case_id`
- `document_id`
- 原始文件路径
- 文书处理任务记录

### 4.2 解析层
负责将 PDF、图片、电子文书等格式统一解析为标准文本结构。

核心能力：
- OCR 识别
- 版面结构解析
- 文书类型识别
- 段落切分
- 页码保留
- 标题、当事人信息区、事实理由区、请求事项区等分区识别

输出对象：
- 文书全文文本
- 文书切片列表
- 文书结构标签
- 页码与段落定位信息

### 4.3 抽取层
负责从切片文本中识别实体、事实片段和候选关系。

建议采用两段式策略：

1. 规则优先
- 当事人信息区规则抽取
- 律师、法定代表人、案号、文书类型等规则抽取
- 常见合同编号、金额、日期、账号尾号等模式抽取

2. 模型增强
- 实体识别
- 事实片段切分
- 关系抽取
- 争议识别
- 主张口径识别

输出对象：
- 候选实体
- 候选事实片段
- 候选关系

### 4.4 归一化层
负责解决“一个人多个名字”“同一公司多个简称”“同一房产不同表述”等问题。

核心策略：
- 规则归一化：统一格式、去噪、标准化
- 多字段匹配：姓名、证件号、统一社会信用代码、账号尾号、合同编号
- 上下文消歧：结合诉讼角色、关联关系、地址、时间
- 人工优先：若已有人工确认映射，优先使用人工结果

输出对象：
- 实体主记录
- 实体别名映射
- 实体合并/拆分建议

### 4.5 映射层
负责将事实片段映射为标准关系三元组，并补充关系属性。

典型处理：
- 将“借给”“出借给”“向其转账借款”统一映射为借贷或支付关系
- 将“其妻子”“系其父亲”“两人为夫妻”统一映射为亲属关系
- 将“系某公司法定代表人”“实际控制该公司”映射为身份控制关系

输出对象：
- 标准三元组
- 关系属性
- 关系状态

### 4.6 来源与评分层
负责为实体、事实、关系挂接文书来源并计算质量信号。

应输出：
- 来源文书标识
- 来源段落
- 来源方
- 抽取方式
- 置信度
- 是否冲突
- 是否待人工确认

### 4.7 存储层
首版建议采用关系型数据库为主、图查询能力为辅的轻量方案。

原因：
- 结构稳定，便于版本管理和审计
- 方便和现有法院业务库集成
- 首版更多是“展示和校核”，不是超复杂图计算

如后续出现复杂路径查询和多跳分析需求，再接入图数据库。

## 6. 版本机制设计

### 6.1 为什么需要版本
案件卷宗会不断补充，抽取模型也会更新，人工校核也会改变结果。如果没有版本机制，系统会出现以下问题：
- 新结果覆盖旧结果，无法追踪变化
- 无法比较重跑前后差异
- 无法还原法官当时看到的图谱状态

### 6.2 版本类型
- 自动抽取版本
- 人工校核版本
- 发布版本

### 6.3 版本规则
- 每次重跑生成新自动版本
- 人工校核基于某个自动版本产生派生版本
- 仅“发布版本”对默认办案视图可见
- 历史版本支持只读查看和差异对比

## 7. 抽取策略设计

### 7.1 规则与模型协同
首版不建议纯模型抽取，应采用如下策略：

| 能力 | 建议策略 |
| --- | --- |
| 当事人信息区抽取 | 规则优先 |
| 文书结构识别 | 规则优先 |
| 金额、日期、编号 | 规则优先 |
| 亲属、借贷、担保等复杂关系 | 模型为主，规则校验 |
| 主张/抗辩/查明口径识别 | 模型为主 |
| 实体消歧 | 规则与打分模型结合 |

### 7.2 大模型使用建议
大模型更适合承担以下任务：
- 长文本事实摘要
- 候选三元组生成
- 关系语义归类
- 冲突事实识别
- 不同文书间的关系整合

不建议直接让大模型“最终定稿”，而应让其输出候选结果，再经过规则约束和人工校核。

## 8. 置信度机制

### 8.1 置信度来源
置信度建议综合以下信号：
- 文书类型权重
- 文本位置权重
- 抽取方法权重
- 多来源交叉印证
- 是否被人工确认

### 8.2 置信度分层
- 高：结构明确、多来源支撑、规则和模型一致
- 中：单一来源但文本表达清晰
- 低：语义不完整、存在歧义或冲突

### 8.3 显示规则
- 高置信关系默认展示
- 中低置信关系可默认折叠或打标签
- 冲突关系必须显式标记

## 9. 人工校核回写机制

### 9.1 回写原则
- 人工结果优先级高于自动抽取
- 自动结果不直接删除，保留审计轨迹
- 校核动作形成新版本，不覆盖底稿

### 9.2 回写类型
- 实体确认
- 实体合并
- 实体拆分
- 关系确认
- 关系修改
- 关系删除
- 来源补充

## 10. 存储方案建议

### 10.1 首版推荐
采用关系型数据库落库，核心表如下：
- `case_graph_version`
- `case_graph_entity`
- `case_graph_entity_attr`
- `case_graph_relation`
- `case_graph_relation_attr`
- `case_graph_source`
- `case_graph_review_log`
- `case_document_chunk`

### 10.2 图查询增强
如后续需要支持多跳路径查询、控制链穿透、资金链分析，可在只读副本上同步至图数据库。

## 11. 任务编排建议

### 11.1 异步任务
以下步骤建议异步化：
- OCR 与文书解析
- 实体抽取
- 关系抽取
- 图谱重跑
- 大模型总结任务

### 11.2 任务状态
建议统一状态：
- 待处理
- 处理中
- 部分成功
- 成功
- 失败
- 已取消

## 12. 安全与审计
- 办案数据应按案件权限隔离
- 图谱查看、导出、校核需记录审计日志
- 敏感信息如身份证号应支持脱敏显示
- 大模型调用需保留任务日志与输入输出摘要

## 13. 首版技术结论
首版最重要的是把“抽取正确率、来源回溯、人工校核、版本可控”四件事做扎实，而不是过早追求复杂图算法。技术实现建议遵循“轻量底座先跑通，复杂图能力后增强”的路线，这样更符合法院业务系统稳妥落地的要求。
